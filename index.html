Can small multimodal LLMs be trained to create complex fine-grained annotations?

An experiment to see if Qwen3-VL (2B, 4B, 8B) can be distilled from GPT-4o annotations to generate captions, scene graphs and keysteps.

Frontier vision models like GPT-4o are remarkably good at understanding sequences of video frames. They can describe what's happening, identify objects and their relationships, and segment activities in time. The catch is cost: it can get expensive to use them to generate annotations for hours of footage.

So as a weekend experiment (which took longer than a weekend), I wanted to try out a hypothesis:

Using GPT-4o once to generate structured fine-grained annotations for a set of domain-specific videos, can we fine-tune much smaller open-source multimodal models do the same job locally?

Data

I chose a set of egocentric (first-person view) PC-building videos for the data. It's a good testbed because the vocabulary is bounded (the objects/components are fairly standard) and the actions are structured (there's a logical build sequence), and since the view is (mostly) egocentric, irrelevant objects/events can be mostly avoided in the annotations.

A quick YouTube search-and-filter process yielded 9 egocentric videos of people building PCs. After trimming some irrelevant segments, the net length of footage is ~8.5 hours. Here are some representative frames (each column is a different video):

Overview

Step 1Frame Extraction

This step samples the frames to be sent for annotation: for each video, k frames every n seconds are sampled using ffmpeg. For this experiment, I set k = 1; n = 5, i.e., 1 frame is sampled every 5 seconds. That's 720 frames/hour of footage.

Step 2GPT-4o Annotations

In this step, each video's sampled frames are encoded as base64 JPEGs and sent to GPT-4o in batches of 16 frames. Three annotation types are produced for every batch:

Caption

A 3-5 sentence description of what's happening in the clip.

Scene Graph

Nodes (objects with types from a pre-defined taxonomy) and edges (spatial and action relationships). Structured, graph-form understanding of the scene.

Keysteps

Temporal segments specifying which step (activity) is happening, and at which frame range.

Prompts

Caption 
SYSTEM:
You are a video understanding assistant. Given a sequence of video frames, produce
a single fluent paragraph (3-5 sentences) describing what is happening in the video.
Focus on the main activity, the agents involved, and any notable objects or state changes.
Be specific and grounded in what is visually present. Do not speculate about off-screen events.

USER:
Below are {n_frames} frames sampled from a {duration_seconds:.1f}-second video
at approximately {fps:.2f} fps. Frame resolution: {width}×{height}.

Describe what is happening in this video.

Respond with a JSON object:
{
  "caption": "<your description here>"
}
Scene Graph 
SYSTEM:
You are a video scene graph generator. Given a sequence of video frames and a fixed
taxonomy, produce a scene graph describing the objects, their relationships, and their
states visible in the video.

RULES:
- Use ONLY the node types, edge types, and state attributes from the taxonomy below.
- Do not invent new vocabulary. If something doesn't fit, pick the closest taxonomy term.
- Assign a unique short ID to each node (n0, n1, n2, ...).
- Include only objects and relationships that are clearly visible in the frames.

DIRECTIONALITY RULES:
- Edge direction is always src → dst.
- Edges marked [src: Hands] MUST have a Hands node as src. Never use a component,
  cable, or tool as src for these edges.
  Correct:   {"src": "n0", "dst": "n1", "type": "screwing"} where n0 is Hands
  Wrong:     {"src": "n1", "dst": "n0", "type": "screwing"} where n1 is Screwdriver
- Edges marked [symmetric] may go in either direction; choose the most natural reading.
- For spatial edges without a [src:] marker, src is the smaller/contained/dependent
  part and dst is the larger/containing/receiving part:
    CPU inserted_in Motherboard  (not Motherboard inserted_in CPU)
    Motherboard mounted_on PC_Case  (not PC_Case mounted_on Motherboard)
    RAM seated_in Motherboard  (not Motherboard seated_in RAM)
    Component inside PC_Case  (not PC_Case inside Component)
- For state edges (is_installed, is_unboxed, is_powered_off, is_unseated, is_secured),
  src is always the component whose state is described; dst is a nearby related node
  (the slot, case, or board it relates to).

TAXONOMY:
{taxonomy_str}

USER:
Below are {n_frames} frames sampled from a {duration_seconds:.1f}-second video.

Generate a scene graph for this video using ONLY the taxonomy terms above.

Respond with a JSON object matching this schema exactly:
{
  "nodes": [
    {"id": "n0", "type": "<node_type>", "label": "<human-readable instance name>"}
  ],
  "edges": [
    {"src": "n0", "dst": "n1", "type": "<edge_type>"}
  ]
}
Keysteps 
SYSTEM:
You are a video activity segmentation assistant. Given a sequence of video frames and a
list of valid activity labels, identify the temporal keysteps in the video.

RULES:
- Use ONLY the labels from the activity labels list below.
- Steps must be in temporal order with non-overlapping frame ranges.
- Start and end frame indices must be within [0, {n_frames_minus_one}].
- Confidence is your certainty that the label and boundaries are correct (0.0–1.0).
- Omit very short transitional frames (< 2 frames) unless clearly a distinct step.

ACTIVITY LABELS:
{activity_labels_str}

USER:
Below are {n_frames} frames sampled from a {duration_seconds:.1f}-second video.
Frame indices run from 0 to {n_frames_minus_one}.

Identify each distinct activity step in temporal order.

Respond with a JSON object:
{
  "keysteps": [
    {
      "label": "<activity_label>",
      "start_frame": <int>,
      "end_frame": <int>,
      "confidence": <float>
    }
  ]
}

Using a pre-defined taxonomy

To ensure that the annotation process doesn't introduce duplicate terminology (e.g., CPU socket and processor slot) and illogical object relationships (e.g., Screwdriver -> holds -> Hand), a domain taxonomy is used, which specifies a controlled vocabulary for node types, edge types, activity labels along with edge directionality constraints. It gets injected directly into the system prompt, and every response is validated against it. I used an LLM (Gemini) to generate a taxonomy based on the most common components and steps in a typical PC building video. Here's what the taxonomy looks like:

Steps 3 + 4Validation & Error Patching

To ensure that there were no bogus annotations (empty strings, non-existing components, illogical object relationships, etc.), there's a basic validation step after the annotations are generated. These were the checks used:

Type Checks
Caption Is a string, at least 10 characters
Scene Graph Node/edge types in taxonomy; edge src/dst reference existing node IDs; src_types constraints satisfied
Keysteps Label in taxonomy; start_frame ≤ end_frame; indices within frame count

Failed batches are written to disk as pretty-printed JSON and then patched manually.

All validation failures which I encountered were scene graph annotations failing edge directionality constraints. These were relatively easy to spot and fix manually.

Step 5Building the Instruction-Tuning Dataset

Once the annotations are validated and patched, they are assembled into JSONL files in the ChatML format commonly used by multimodal instruction-tuning frameworks.

Data splits

The data is split into train / val / test (80 : 10 : 10 ratio) at the video level instead of the example level, since if frames from the same video appear in both train and test, the model can trivially memorize visual context and inflate performance. Step 6Training Qwen3-VL LoRA adapters

I chose LoRA training since I wanted short training runs for this experiment. For the base models, I picked Qwen3-VL with multiple parameter variants (2B, 4B, 8B). Qwen3-VL is one of the recent strong open-weights multimodal LLMs available, and using the different parameter variants makes it easy to reason about performance scaling over parameter size.

The fine-tuning was done using the axolotl LLM training framework. I trained only on the language model layers, assuming the vision layers are already adequate for this domain. Here is a sample config file which I had used for the 4B model: 

base_model: Qwen/Qwen3-VL-4B-Instruct
output_dir: outputs/qwen3vl_4b/

processor_type: AutoProcessor
chat_template: qwen2_vl          # Qwen3-VL reuses qwen2-vl template
skip_prepare_dataset: true
remove_unused_columns: false
sample_packing: false

datasets:
  - path: outputs/dataset_normalized/train.jsonl
    type: chat_template
test_datasets:
  - path: outputs/dataset_normalized/val.jsonl
    type: chat_template

sequence_len: 3904

# Training
bf16: true
fp16: false
tf32: false
torch_dtype: bfloat16
flash_attention: true
gradient_checkpointing: true
gradient_checkpointing_kwargs:
  use_reentrant: false
requires_inputs_embeds_grad: true
num_epochs: 8
warmup_ratio: 0.03
lr_scheduler: cosine
optimizer: adamw_torch_fused

# LoRA language model layers only (vision tower frozen)

Step 7Evaluation

Since we do not have manually curated annotations, a key assumption I make is that the GPT-4o annotations are human-quality ground truth (which is a really strong assumption). I evaluated the models using a combination of hard (objective) metrics and LLM-as-a-Judge API calls. I chose GPT-5.5 as the Judge because it is currently the strongest OpenAI model and required minimal changes to existing code. I describe the metrics below:

Metric Name Annotation Type Metric Type Description
factual_accuracy Caption LLM-as-a-Judge Evaluates if the described objects and actions are actually correct on a 1-5 scale.
completeness Caption LLM-as-a-Judge Measures if the prediction covers the main events from the ground truth on a 1-5 scale.
specificity Caption LLM-as-a-Judge Checks if the text is grounded in specific visible details rather than being generic (1-5 scale).
node_type_f1 Scene Graph Hard Metric A token-level multiset F1 score measuring how well the predicted object types match the ground truth.
edge_type_f1 Scene Graph Hard Metric A token-level multiset F1 score measuring how well the predicted relationship types match the ground truth.
semantic_quality Scene Graph LLM-as-a-Judge Evaluates if the predicted nodes and edges capture the same overall scene meaning as the ground truth on a 1-5 scale.
label_f1 Keysteps Hard Metric A multiset F1 score evaluating how well the predicted activity labels match the actual labels.
mean_iou Keysteps Hard Metric Calculates the average temporal overlap (Intersection over Union) of the predicted step frame ranges versus the ground truth.
ordering_quality Keysteps LLM-as-a-Judge Evaluates the temporal coherence and correct chronological ordering of the predicted steps on a 1-5 scale.

Observations

1. Across all annotation types, the fine-tuned models (-ft) consistently outperform their zero-shot (-zs) counterparts on both hard metrics and LLM-as-a-judge evaluations.

2. For scene graph generation, zero-shot performance is near zero, suggesting that this annotation type is the hardest for the models to learn.

3. There were no big gains on increasing parameter size from 4B to 8B. The 4b-ft model matches the 8b-ft model on most metrics, and is noticeably better for scene graph generation.

Final Thoughts

1. From a preliminary look, it does seem the smaller Qwen3-VL models are capable of learning complex fine-grained spatial (scene graphs) and temporal (keysteps) annotations. I need to do a deeper dive to see whether this can reach levels of practical utility.

2. The biggest caveat of this experiment is that the evaluation is done on a measly 49 test records per annotation type. This is also likely the reason for the high variance seen in the score plots.

3. As mentioned earlier, I assume GPT-4o annotations as absolute ground truth. These are likely to have errors and ideally need to be curated further before usage as training data.

4. Using another GPT model as the Judge might have introduced some bias in the LLM-as-a-Judge scores. It is probably best to use a strong non-OpenAI model for a more impartial evaluation.
